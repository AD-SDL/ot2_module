# AGENTS.md/CLAUDE.md

This file provides guidance to AI coding agents when working with code in this repository. Note that AGENTS.md and CLAUDE.md are symlinked.

## Overview

`ot2_module` is a MADSci node module for Opentrons liquid handlers (both OT-2 and Flex). It exposes a FastAPI REST server that integrates with the MADSci framework, and communicates with the physical robot over its built-in HTTP API (port 31950).

See `MADSci/CLAUDE.md` (or `MADSci/AGENTS.md`) for MADSci framework architecture, node module patterns, and conventions this module follows.

## Configuration

Configuration uses pydantic-settings with `NODE_` prefix. Sources (in priority order): environment variables → `.env` → `settings.yaml`. Key fields:

- `NODE_OT2_IP` — IP address of the Opentrons robot (required)
- `NODE_URL` — URL the node advertises (default `http://127.0.0.1:2000/`)
- `NODE_NODE_NAME` / `NODE_NODE_ID` — node identity

See `docs/Configuration.md` for the full reference (auto-generated by `pydantic-settings-export`). The `.env.example` file lists all available env vars with their defaults.

## Architecture

### Entry Point
`src/ot2_rest_node.py` — `OT2Node(RestNode)` is the MADSci node implementation. It:
- Extends `RestNodeConfig` with `ot2_ip: Optional[str]`
- Connects to the robot via `OT2_Driver` on startup using `self.config.ot2_ip`
- Registers one action: `run_protocol` (decorated with `@action`)
- Implements `startup_handler`, `shutdown_handler`, `state_handler`, and control methods (`pause`, `resume`, `cancel`, `safety_stop`, `reset`)
- Uses `self.node_info.node_name` / `self.node_info.node_id` for identity (not a definition file)

### Node Identity (settings-based pattern)
Node identity (`node_name`, `node_id`, etc.) is configured via `settings.yaml` or env vars — there are no separate definition files. The `NodeInfo` object is built automatically from the config by `NodeInfo.from_config()` in the MADSci base class.

### OT2 HTTP Driver
`src/ot2_interface/ot2_driver_http.py` — `OT2_Driver` wraps the Opentrons REST API:
- `transfer(protocol_path)` → uploads a `.py` protocol file, creates a run, returns `(protocol_id, run_id)`
- `execute(run_id)` → sends a `play` action, polls until run completes (SUCCEEDED/FAILED/STOPPED)
- `pause(run_id)`, `resume(run_id)`, `cancel(run_id)` → lifecycle control
- `get_run_log(run_id)` → fetches run data + commands (up to 1000)
- `compile_protocol(config_path, ...)` → delegates YAML→Python compilation to Protopiler (skips if `.py` already)

`OT2_Config` (in `src/ot2_interface/config.py`): `ip`, `port` (default `31950`), `model`.

### Protopiler
`src/ot2_interface/protopiler/` — Compiles YAML protocol configs into Opentrons Python protocol files using string templates.

**Config structure** (`ProtocolConfig` in `protopiler/config.py`):
- `equipment`: list of `Labware` (deck slot + name + optional alias) and `Pipette` (name + mount)
- `commands`: list of typed command objects (`Transfer`, `Multi_Transfer`, `Mix`, `Move_Labware`, `Temperature_Set`, `Deactivate`, `Replace_Tip`, `Clear_Pipette`, `Move_Pipette`, `Ninetysix_Transfer`)
- `metadata`: protocol name, author, description, apiLevel
- `resources`: optional list of external files (xlsx) for data-driven transfers

**Transfer command** fields include `source`, `destination` (e.g. `"source:A1"` using labware aliases), `volume`, `aspirate_clearance`, `dispense_clearance`, `mix_cycles`, `blow_out`, `drop_tip`, `return_tip`. List fields must all be the same length.

Templates live in `protopiler/protocol_templates/*.template`. Example configs are in `protopiler/test_configs/`.

### Parameter Substitution in `run_protocol`
The `run_protocol` action accepts a `parameters` dict. Before execution, occurrences of `$key` in the protocol `.py` file are replaced with the corresponding values. This is a simple string substitution (not Jinja2).

### Protocol Execution Flow
1. MADSci sends `run_protocol` action with a protocol `.py` file and optional parameters
2. `run_protocol` substitutes parameters into the file, then calls `execute()`
3. `execute()` calls `ot2_interface.transfer()` (upload + create run) then `ot2_interface.execute()` (play + poll)
4. On success, `get_run_log()` is called and optionally used to update MADSci resources

### Docker Deployment
- Base image: `ghcr.io/ad-sdl/madsci`
- Published image: `ghcr.io/ad-sdl/ot2_module`
- Install: `uv pip install --python ${MADSCI_VENV}/bin/python -e /home/madsci/ot2_module`
- Uses `network_mode: host` (required to reach OT-2 on the local network)
- Volumes: `settings.yaml` and `.madsci/` are bind-mounted (no definition files)
- `USER_ID` / `GROUP_ID` build args control file ownership

### Tooling
- `pyproject.toml` configures `pydantic-settings-export` to generate `docs/Configuration.md` and `.env.example` from `OT2NodeConfig`
- The `pydantic-settings-export` pre-commit hook regenerates these files when `ot2_rest_node.py` changes
- Ruff is configured with `target-version = "py310"` and pydocstring requirements (`D1xx`)
